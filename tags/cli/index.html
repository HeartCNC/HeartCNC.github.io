从0到1，基于tinypng压缩图片的CLI工具

2022年5月20日

懒癌发作，制作一个压缩图片的CLI工具吧！

Gitlab 地址：https://gitlab.inrice.top/longxiaojian/iojs

使用方法​

拉取源代码到本地，并安装依赖，通过全局软链接后，后面如果有新代码的话，直接拉取，就可以自动更新使用。自己有新的想法，或者开发一项便利工具，也可以直接在代码中开发，实时更新到全局。

CLI全局软链接​

npm link
note

npm link - 创建软链接，类似于快捷方式，相当于全局安装，比如

本地开发包 module

module
npm link

本地项目 project

project
npm link module

此时，依赖 module 被安装到 project 项目中

npm unlink - 移除当前包全局软链接

npm unlink module - 当前项目移除包软链接

参数​

  • -c，--compress 执行压缩

  • -i，--init 初始化配置文件，当项目是二次开发，其中的图片可能已经被压缩过了，因此用该命令来初始化配置，避免不必要的处理。与 -c 互斥，优先处理 -i

  • -d，--dir 指定目录，默认是项目根目录，自动忽略 node_modules 中的图片

  • -s，--size 指定处理文件大小的最小范围，单位：k，默认是0

示例​

  • 对当前文件夹下的所有图片压缩
io image -c
  • 压缩指定文件夹的图片
io image -c -d src/assets
  • 压缩 src 文件夹下超过 20k 的图片
io image -c -d src -s 20

编写核心代码​

实现过程​

  1. 读取图片路径 filepaths
  2. 读取本地配置 lockfile，根据文件的修改时间，过滤出待压缩的列表 paths
  3. 上传图片到tinypng，下载图片到本地
  4. 更新配置文件

imagemin-lock.yaml 配置文件中存储的是图片修改时间戳，key为文件相对于项目的相对路径，value为文件修改时间戳

利用配置来比对本地文件，是为了防止图片重复压缩

辅助方法​

utils/index.js
const path = require('path')
const fs = require('fs')
const { glob } = require('glob')
const { workspaceDir } = require('./shared')

/**
 * 相对项目根目录的绝对路径
 */
exports.rootPathResolve = function (...p) {
  return path.join(workspaceDir, ...p)
}

/**
 * 读取本地文件路径列表
 */
exports.readfile = async function (baseUrl, rule) {
  return new Promise((resolve, reject) => {
    const p = path.join(baseUrl, `**/*.@(${rule})`)
    glob(p, {}, function (err, files) {
      err ? reject(new Error(err)) : resolve(files)
    })
  })
}

/**
 * 写文件到本地
 */
exports.writeBlob = function (path, contents) {
  return new Promise((resolve, reject) => {
    fs.writeFile(path, contents, 'binary', (err) => {
      err ? reject(err) : resolve()
    })
  })
}

指令执行​

scripts/image.js
const fs = require('fs')
const yaml = require('js-yaml')
const tinypng = require('../utils/tinypng')
const { rootPathResolve, writeBlob, readfile } = require('../utils')
const chalk = require('chalk')
const lockfile = rootPathResolve('imagemin-lock.yaml')
const maxSpread = 10
const spreadPool = []

function getRelativePath(base, path) {
  return path.replace(new RegExp(`^${base}/`), '')
}

module.exports = async function (options) {
  // 根目录前缀
  const prefix = rootPathResolve('').replace(/\\/g, '/')
  // 输入目录
  const inputUri = rootPathResolve(options.dir || '')
  // 读取所有图片路径
  const filepaths = await readfile(inputUri, 'png|jpg|jpeg|webp')

  // 不存在则创建一个文件
  if (!fs.existsSync(lockfile)) {
    fs.writeFileSync(lockfile, '')
  }

  const minCompressSize = Number(options.size)
  const content = fs.readFileSync(lockfile)
  // 旧配置
  const mtimeOld = yaml.load(content) || {}
  // 最新配置
  const mtimeCur = {}

  // 开始比较
  filepaths.forEach(item => {
    // 文件信息
    const stat = fs.statSync(item)
    const path = getRelativePath(prefix, item)
    
    if (stat.size < minCompressSize * 1024) {
      return
    }
    mtimeCur[path] = stat.mtimeMs
  })

  // 根据比较之后需要压缩的路径列表
  const paths = []

  // 获取差异文件路径
  for (const k in mtimeCur) {
    const mtimeMsCurItem = mtimeCur[k]
    const mtimeMsOldItem = mtimeOld[k]
    if (!mtimeMsOldItem || mtimeMsOldItem !== mtimeMsCurItem) {
      paths.push(k)
    }
  }

  let total = 0
  let failPath = []
  const len = paths.length

  const updateLock = () => {
    fs.writeFileSync(lockfile, yaml.dump(mtimeOld))
    if (total === len) {
      if (failPath.length) {
        console.log(chalk.red(failPath.join('\n')))
        console.log(chalk.red(`Fail: ${failPath.length}.`))
      }
    }
  }

  const updatePool = (index) => {
    if (queue.length <= 0) {
      return
    }
    spreadPool[index] = queue.shift()
    spreadPool[index](index)
  }

  const queue = paths.map((item) => {
    return async (index) => {
      const path = getRelativePath(prefix, item)
      try {
        const contents = await tinypng(item)
        await writeBlob(item, contents)
        mtimeOld[path] = fs.statSync(item).mtimeMs
        console.log(`${item}: success ${++total}/${len}`)
      } catch (error) {
        failPath.push(path)
        console.log(chalk.red(error.message || error))
        console.log(`${item}: fail ${++total}/${len}`)
      }
      updateLock()
      updatePool(index)
    }
  })

  spreadPool.push(...queue.splice(0, maxSpread))
  spreadPool.forEach((item, index) => item(index))
}

注意事项​

  1. 为了避免在操作过程中，因为网络等问题，让配置未及时更新，所以每次处理完成一张图，都更新本地配置文件。因为写文件频繁，所以在源代码中的 updateLock 加了防抖。

  2. 同时并行处理上传多张图片时，由于接口的限制会报 Too many files uploaded at once.，因此代码中利用了并发池来控制请求量，来提高容错率。

  3. 如果碰到长时间没有反应，可能请求崩了，很正常，可以直接 Ctrl + C 结束进程，重新运行即可。

  4. 如果碰到多次 ETIMEDOUT 可能是 tinypng 崩了，很正常，等它一会儿吧，过会儿重新运行即可。

  5. 由于可能存在文件删除的情况，所以配置会移除不存在的路径，以下操作需要注意

io image -c -d src/assets

此时配置中是 src/assets 的

io image -c -d src/assets/img

此时配置会覆盖是 src/assets/img
